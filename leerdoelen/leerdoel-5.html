Code review en Feedback sessie

Hoe wordt het gedaan?

Code review is een proces waar programmeurs elkaar codes bespreken en beoordelen, of efficient en gestructureerd geschreven is. Het voorkomt dat je bugs in je project krijgt. Ik krijg vaak code review wanneer ik een opdracht of functionaliteit afrond bijvoorbeeld de html structuur, of vormgeving van de pagina's. Ik plant de vergadering in en ik ga samen met de programmeur naar mijn werk kijken. De frontend developer gaat het figma ontwerp kijken en daarna het geprogrammeerde prototype. Hij benoemt de feedbackpunten, daarna ga hij naar mij code kijken, soms geef hij mij tips hoe je de code beter kan schrijven.

Feedback moment

Het is ook een moment waar ik de kans krijgt om mijn code te onderbouwen. Dus ik ga vertellen waarom ik dit heb gedaan en waarom dat een goede keuze is. Vooral bij mijn eindproject moest ik veel gaan onderbouwen want het ging over een website waar ze eventueel online ga zetten. Hiermee ben ik beter in geworden om mijn keuzes te onderbouwen.

Voordelen van een code review

  • Er wordt naar me code gekeken
  • Ik krijg feedback op mijn code
  • Ik leer nieuwe trucjes die ik kan gebruiken
  • Ik kan mijn keuzes op bepaalde functionaliteiten onderbouwen
  • Ik kan van de programmeurs leren

Observeren hoe je aan klanten presenteert

Wireframes aan klanten presenteren

Als onderdeel van mijn observatieopdracht wilde ik weten hoe je prototypes aan klanten presenteren. Op een dag kreeg de kans om bij te zijn de wireframe presentatie voor de opdrachtgever stichting Prixma. Stichting Prixma wilde een nieuwe website bouwen en de UX designer heeft daardoor de wireframe van de Homepagina en een sitemap presenteren.

Ik heb gevraagd om bij de presentatie aanwezig te zijn om het project aan klanten te observeren, zodat ik kan zien hoe deze presentaties worden uitgevoerd. Hierdoor heb ik veel kunnen leren.

Hoe is het gegaan?

  • Voor online gesprekken met klanten heb worden altijd met Google meet gewerkt.
  • Projectmanager start met de introductie:
    • Iedereen voorstellen
    • Doel van deze gesprek namelijk het presenteren van de homepagina, wireframe en de sitemap. Daarna woord geven aan de UX-designers
  • Wat eerste wat gebeurde was dat de UX-designer deel scherm en laat aan iedereen de Figma bestand zien liep door de sitemap heen en onderbouwde bepaalde keuzes die ze heeft gemaakt en tussendoor vraag zij feedback aan de opdrachtgever of vragen voor bepaalde richtingen
  • Zij legde wel van tevoren uit wat de sitemap inhoud - Structuur hoe je de pagina navigeert
  • Wanneer de opdrachtgever een feedback punt benoemt, wordt het meteen in de Figma bestand naar de lay-out element genoteerd.
  • Wanneer alles duidelijk waren, ging de UX-designer verder met het wireframe homepagina. Daar heeft de projectmanager uitgelegd wat ze gaan verwachten bij een wireframe heen. Hier werd de homepagina doorgenomen en heeft ze wat punten onderbouwde.
  • Wanneer we helemaal klaar was, legde de projectmanager uit wat de vervolgstappen na deze gesprek waren. En verzoekte de opdrachtgevers om een email sturen met allerlei feedback over de wireframe. Voordat krijgen ze wel een pdf van die sitemap en wireframe. Met de feedback wordt een. Nieuwe versie gemaakt door de UX designer.

Handige tips

  • Begin presentatie met een duidelijke introductie, waarin deelnemers worden voorgesteld.
  • Begin presentatie met een duidelijke introductie, om te vertellen waarover het presentatie overgaat.
  • Gebruik online tools zoals google meet voor online vergadering
  • Verklaar belangrijke termen en concepten, zoals de betekenis van de sitemap en de structuur van de pagina-navigatie. Zorg ervoor dat alle betrokkenen een goed begrip hebben van de gebruikte terminologie.

4stagiairs eindpresentatie

Presenteren voor de team van go2people

Wanneer ik klaar was met mijn eindproject , kreeg ik de mogelijkheid om mijn project te gaan presenteren aan de hele team. Ik heb daardoor een powerpoint presentatie voorbereid en mijn presentatie houden. Er waren ook sommige collega's die online keken. De powerpoint presentatie heb ik via Figma gemaakt met dezelfde huisstijl van de website zelf. Het was makkelijker om de kleuren te overnemen.

Tijdens mij presentatie heb ik de hele proces laten zien. De ideegeneratie, mijn schetsen, Figma prototypes en de wordPress website zelf. Ik heb alle web elementen kunnen uitleggen en mijn keuze onderbouwd. Het werd makkelijker door al die code review die ik had met mijn stagebegeleider

Reflectie op het leerdoel

Gedurende mijn stage heb ik me gericht op het verbeteren van mijn presentatievaardigheden en het effectief ontvangen van feedback. Dit was een belangrijk leerdoel, aangezien communicatie en samenwerking cruciaal zijn in een professionele omgeving.

De laatste twintig weken heb ik veel kunnen leren en beter worden in het presenteren van mij code. Het is dankzij de regelmatige feedback sessie en de code review met de frontend developer. Hierbij heb ik de kans gehad om mijn beslissingen en keuzes te onderbouwen. Dit proces bood niet alleen de mogelijkheid om mijn technische vaardigheden te delen, maar ook om waardevolle feedback te ontvangen die mij in staat stelde mijn werk te itereren.

Een hoogtepunt van mijn leerproces was de presentatie van mijn WordPress-website aan het gehele Go2People-team, waarbij ik mijn werk aan collega's van verschillende expertise kon presenteren. Dit gaf me de mogelijkheid om mijn ontwikkelde vaardigheden in de praktijk te brengen en te tonen aan een breder publiek.
